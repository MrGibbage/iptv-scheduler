# iptv-scheduler — Plan

## Overview

A companion web service to [iptv-recorder](/srv/iptv-recorder/PLAN.md) — the content-intelligence layer that decides *what* to record. Where iptv-recorder is deliberately dumb (mechanical scheduling primitives, no concept of "show" or "episode"), iptv-scheduler owns EPG ingestion, rule-based filtering, automatic discovery, and duplicate detection, then drives iptv-recorder's API to actually make recordings happen — the same API any other client (like Laomedeia) could use directly.

Runs on docker-server or smavm — likely alongside iptv-recorder, host TBD.

## Secrets Handling

This repo is **public**, same as iptv-recorder. Applies here too — this service will hold at least an API key for calling iptv-recorder, and possibly credentials for a secondary EPG source:

- No real credentials, API keys, tokens, or `.env` files with live values are ever committed. Real config lives outside the repo or in a git-ignored file; only a placeholder `.env.example`/template is tracked.
- `.gitignore` must cover the real config/secrets file(s) from day one of actual implementation.
- Real credentials/keys for this project shouldn't be pasted into Claude chat sessions either — describe config by shape/placeholder, not real value.

## Tech Stack

Decided (mirrors [iptv-recorder](/srv/iptv-recorder/PLAN.md), which hit this same open question first):
- **Backend:** Node.js + TypeScript, Fastify for the HTTP layer.
- **Database:** SQLite via Drizzle ORM + better-sqlite3.
- **UI:** React + Vite SPA — this also resolves the "own API/UI vs. headless" open question below: iptv-scheduler exposes its own API and a web UI, it isn't purely a background service. Dev-mode Vite proxies `/api` to the Fastify server.
- **Package manager:** pnpm (workspace with `server/` + `web/` packages).
- **Host:** docker-server (alongside iptv-recorder).

**Repo layout:** same shape as iptv-recorder — pnpm workspace root + `server/` + `web/` packages. `.env`/`.env.example` live in `server/`, not the repo root (see iptv-recorder's Tech Stack note for why — `dotenv/config` resolves relative to `process.cwd()`, which is the package dir under `pnpm --filter`). Scaffolded and verified booting end-to-end (health check + Vite API proxy) 2026-07-19. No real DB tables yet — `schema.ts` is a placeholder pending the rule-schema design (TODO2).

## EPG Source

Decided: guide data comes only from the Xtream provider's own EPG endpoint, no secondary XMLTV source for now. Revisit only if that data proves inaccurate/incomplete in practice.

This is entirely iptv-scheduler's concern — iptv-recorder has and needs zero EPG awareness. iptv-scheduler ingests/caches the Xtream EPG, matches rules against it, resolves a rule match down to a concrete channel + time slot, and calls iptv-recorder's plain `POST /recordings` (`provider_id` + `channel_id` + start/end time) exactly like any other client would. No title, episode, or guide data ever crosses that API boundary.

## Relationship to iptv-recorder

Division of responsibility (mirrors the note in iptv-recorder's plan):

- **iptv-recorder:** mechanical scheduling (one-off + recurring), provider credential/account management, concurrent-stream enforcement, storage, retention. No content awareness.
- **iptv-scheduler (this project):** EPG ingestion, filter rules, discovery, duplicate detection against content identity (show/episode) — decides *what* to record, then calls iptv-recorder's schedule endpoint to make it happen.

iptv-scheduler is a client of iptv-recorder, not a component of it. It must use the same public API surface as any other client — no special/private integration.

Because iptv-recorder has no concept of duplicate content, iptv-scheduler is responsible for actively cancelling/skipping a specific recurring occurrence through the recorder's API if it determines that occurrence would be a duplicate (e.g. a rerun already recorded elsewhere).

## Goals (draft feature list)

**Discovery & rules**
- Series/season-pass style rules — "record all new episodes of X," matched against EPG data rather than a fixed time slot
- Keyword, category, and channel-based rules (e.g. "record anything tagged 'documentary' on these channels")
- Exclude filters — skip reruns/repeats (original air date vs. current air date, where EPG data supports it), skip by keyword
- Priority ranking between rules, used for conflict resolution

**Duplicate detection & conflict resolution**
- Dedup against already-recorded or already-scheduled content by episode identity (title + season/episode, or title + air date as a fallback given inconsistent provider EPGs)
- When a candidate recording would exceed a provider's concurrent-stream limit, iptv-recorder hard-rejects — iptv-scheduler needs to resolve this itself (preempt a lower-priority scheduled recording, or skip and notify) rather than just surfacing the raw rejection
- Cross-provider awareness if the same channel exists on more than one configured provider — prefer the one with headroom

**EPG / guide management**
- Ingest and cache XMLTV/provider guide data
- Merge/dedupe guide data if a second EPG source is ever added
- Search across the guide — this is what rule-matching runs against

**Notifications & visibility**
- Recording started/completed/failed/skipped-due-to-conflict
- Low disk space or retention-policy warnings (surfaced from iptv-recorder's config)
- Provider auth failure alerts

**Library / retention intelligence**
- "Keep last N episodes" per series, watched-status-aware deletion — sits on top of iptv-recorder's raw TTL/cap retention policy, doesn't replace it
- History log of past recordings with outcome and reason (useful for debugging missed recordings)

**Manual override**
- Browse the guide and force a one-off recording outside of any rule, via iptv-recorder's normal API — same path any other client would use

**Explicitly deferred (not v1)**
- Metadata enrichment (TMDB/TVDB posters/descriptions) — UI polish, not core function

## Rule Schema

Decided 2026-07-21: a rule is a set of **combinable filters, not a discriminated type**. The "series," "keyword," "category," and "channel" rule flavors from the Goals list above are all just different combinations of the same optional filter columns on one `rules` table, ANDed together against an EPG program — not separate tables or a `type` enum with per-type shapes. This also covers combinations the Goals list already implies without saying so (e.g. "record anything tagged 'documentary' on these channels" is a category filter *and* a channel filter together), and keeps the future matching engine to one code path instead of one per rule type.

**Renamed "genre" → "category" 2026-07-22, direct user correction: "category" is the actual IPTV/Xtream-domain term** (matches the Xtream API's own `category_name`/`category_id`, see `../epg/xtream.ts`) — "genre" was this codebase's own wrong guess, not something any provider or spec actually calls it. Applied throughout: the `rules.categories`/`channels.category`/`epg_programs.category` columns (renamed via a hand-verified `ALTER TABLE ... RENAME COLUMN` migration, not drizzle-kit's default destructive drop+recreate — see EPG Ingestion below for why that mattered), the matcher, every API field, and the UI. All mentions below use the corrected name regardless of which date the surrounding work was actually done.

**Implemented 2026-07-21** (`server/src/db/schema.ts`):
- `rules` table — positive filters `seriesTitle` (substring match, case-insensitive planned — deliberately not exact, since EPG titles vary by provider/episode), `keywords` (JSON string[], `keywordMatchMode: "any" | "all"`), `categories` (JSON string[]), `channelIds` (JSON string[], opaque per-provider like iptv-recorder's `recordings.channel_id`). All are nullable/optional and AND together; a CHECK constraint (`rules_has_positive_filter`) rejects a rule with none of the four set, since that would match every program in the guide. Verified against a scratch SQLite DB: a rule with `seriesTitle` set inserts fine, a rule with all four null fails the CHECK.
- Exclude filters: `excludeKeywords` (JSON string[]) and `excludeReruns` (boolean) — the latter is stored but a no-op until EPG ingestion (TODO3) decides whether/how original-air-date data is exposed after caching.
- `priority` (integer, default 0, higher wins) — carried on the rule now so rules can be ranked from day one, ahead of the conflict-resolution logic (TODO3) that will actually consume it.
- `enabled` (boolean, default true) — pause without delete, mirrors iptv-recorder's provider `enabled` pattern.
- ~~A minimal local `providers` table...~~ **Superseded 2026-07-22** — see EPG Ingestion below: there is no local `providers` table at all now. `rules.providerId` is a plain opaque integer (iptv-recorder's own provider id), not a local FK.

**Execution note (not yet built):** a rule match produces a *one-off* `POST /recordings` call to iptv-recorder, not a recurring one — iptv-recorder's recurrence primitive is a fixed day/time-of-week pattern (see its PLAN.md, "Recurring occurrence materialization"), which doesn't fit EPG-driven matches where a new episode's air time varies week to week.

## Rule Matching

**Implemented 2026-07-22** (`server/src/rules/matcher.ts`, `server/src/routes/rules.ts`), once real EPG data existed to match against: full CRUD for `rules` (`POST`/`GET`/`GET :id`/`PUT`/`DELETE /rules`, hard delete — rules are config, not history, unlike iptv-recorder's soft-deleted resources) plus `GET /rules/{id}/matches` to preview what a rule currently matches, ahead of duplicate detection or actual scheduling (both still separate, undesigned work — TODO2/TODO3).

- **Matching strategy: SQL narrows, JS filters.** `providerId`/`channelIds` (if set) become a SQL `WHERE` against `epg_programs` first — cheap, and channelIds especially can cut the candidate set down a lot. `seriesTitle`/`keywords`/`categories`/exclude filters then run in JS against that candidate set, since they're free-text substring checks or JSON-array membership that aren't worth forcing into SQL (no JSON1 querying) at the data volumes seen so far (~100k rows for one real provider).
- **`seriesTitle`/`keywords`: case-insensitive substring**, matching the schema's own documented intent. **`categories`: case-insensitive exact match, not substring** — a category is a discrete name a rule would pick from a list, not free text.
- `channelIds` is applied once, at the SQL-narrowing step — not re-checked in the JS filter, since re-deriving it there would be redundant.
- `excludeReruns` is wired up (compares `originalAirDate` to `startTime`) but is a live no-op today, consistent with EPG Ingestion above — no tested provider populates `originalAirDate`.
- Validation: "at least one positive filter" (the same rule the `rules_has_positive_filter` CHECK enforces at the DB level) is also checked in the route handler, against the *merged* result for `PUT` — a partial update clearing a rule's only filter is rejected with a clear 400, not left to surface as a raw SQLite constraint error.
- Verified end-to-end against the real cached EPG data (98,697 real programmes, see EPG Ingestion): a `seriesTitle: "Big Bang Theory"` rule returned exactly 729 matches, matching a raw `COUNT(*)` on the same title exactly; a `categories: ["US| BIBLICAL/RELIGIOUS"]` rule returned 435, likewise matching a raw count; adding `excludeKeywords: ["Halloween"]` to the first rule dropped it from 729 to 724, correctly excluding the 5 Halloween-episode airings; both validation guards (no-filter create, filter-clearing update) returned 400 with a clear message instead of a raw DB error; delete returns 204 and a subsequent `GET` 404s, deleting an unknown id 404s. All test rules removed after.

## EPG Ingestion

Decided 2026-07-22, resolving the "Provider credentials for EPG access" open question from the Rule Schema section: **iptv-recorder stays the single source of truth for provider accounts and credentials.** No separate `providers` table here, no duplicate copy of Xtream credentials, and no third "account manager" service — a third service was considered and rejected as unnecessary complexity for a two-service setup. iptv-scheduler is a pure client of iptv-recorder's provider data, same as it already is for scheduling.

- **iptv-recorder side:** `GET /providers/{id}/connection` — a new, deliberate exception to iptv-recorder's "credentials redacted in every response" rule, returning raw `{ baseUrl, username, password }`. Reasoned acceptable because Xtream URLs already embed credentials as standard practice, so this isn't a new class of exposure. See [iptv-recorder's PLAN.md](/srv/iptv-recorder/PLAN.md) API Design → Providers and Credentials Model for the full implementation, done 2026-07-22.
- **Provider Xtream credentials: fetch-fresh, not cached.** iptv-scheduler calls `GET /providers/{id}/connection` each time it needs to reach a *provider's* Xtream endpoint, rather than storing what it gets back. Trades a small coupling (an EPG refresh silently stalls if iptv-recorder is down at that exact moment) for not needing its own at-rest encryption story for a second copy of that secret. Given both services run on the same host for the same operator, and a recorder outage already stops actual recording anyway, this isn't considered a meaningful new failure mode.
- **iptv-scheduler's own connection *to* iptv-recorder is different — that one is DB-backed, not fetch-fresh, decided 2026-07-22.** There's nowhere else to fetch it from (it's the thing that makes fetching anything else possible), so it has to be stored somewhere; `server/.env`-only was the alternative, rejected in favor of a web UI per direct user request ("let's get all the scaffolding built to the point where I can set up an API key... the final app will do this in a UI settings page"). See "Settings: DB-backed connection config" below.
- **Refresh mechanism: in-process interval, not OS cron** — mirrors why iptv-recorder's own scheduler engine isn't OS cron either (see its PLAN.md, "Scheduler engine"): no external moving parts, and the interval is itself just app config. Default `EPG_REFRESH_INTERVAL_MS` = 6 hours (`server/src/config.ts`) — deliberately much coarser than iptv-recorder's 30s scheduler tick, since guide data doesn't change minute-to-minute.
- **Responsiveness:** the rule matcher and any future UI only ever read the local `epg_programs` cache table, never a live Xtream fetch — that's what actually guarantees a responsive app regardless of upstream provider latency, independent of how often the background refresh runs (user requirement, 2026-07-22: "most important to keep the app responsive").

**Scaffolding implemented 2026-07-22:**
- `server/src/epg/refresh.ts` + `server/src/epg/index.ts` — `runEpgRefresh()` lists enabled providers and resolves fresh connection info for each; wired into server boot/shutdown in `src/index.ts` via `startEpgRefresh()`/`stopEpgRefresh()`, same pattern as iptv-recorder's `startScheduler()`/`stopScheduler()`.
- `epg_programs` table (`server/src/db/schema.ts`) — providerId/channelId (both opaque, no local FK, same pattern as `rules`), title/description/category, startTime/endTime (CHECK: end after start), `originalAirDate` (nullable, feeds `rules.excludeReruns`), `fetchedAt`.

**Real parser implemented 2026-07-22, against a real provider ("sonix," 4518 channels), once the Settings page made a live connection available to probe against:**
- **Bulk `xmltv.php` dump, not a per-channel `get_short_epg` loop** — decided after probing both live. The per-channel JSON API (`player_api.php?action=get_short_epg&stream_id=…&limit=N`) is simple (keyed directly by `stream_id`) but would mean one HTTP request per channel — 4518 of them for this provider alone, every refresh cycle. `xmltv.php` returns everything in one request (verified: 30MB, ~101k `<programme>` entries, ~2 days of guide data across ~1900 channels with EPG at all) — far more provider-friendly, and gives materially deeper look-ahead per channel than a capped `limit=N` short-EPG call would. Costs a real XML parse (`fast-xml-parser`, new dependency) and an id-translation step (below) that the per-channel API wouldn't have needed — accepted as the right tradeoff at this scale, not premature optimization.
- **Two different channel-id spaces, discovered by probing, not assumed:** `<programme channel="X">` in the XMLTV dump refers to the provider's `epg_channel_id`, a completely different identifier from `stream_id` — e.g. one real channel had `stream_id: 67588111` but `epg_channel_id: "1518734"`, and the XMLTV file's `<programme channel="1518734">` entries are the ones that actually belong to it. `epg_programs.channelId` stores `stream_id` (what iptv-recorder needs to actually record), so ingestion fetches `get_live_streams` (unfiltered — confirmed live it returns every channel in one call, no need to loop by category) to build an `epg_channel_id → stream_id` translation map before the XMLTV entries mean anything. `server/src/epg/xtream.ts` (the two JSON calls) and `server/src/epg/xmltv.ts` (the XML parse + XMLTV-datetime parsing, e.g. `"20260721053700 +0000"`) split along this line.
- **Category comes from the channel's own category, not the EPG listing** — confirmed live that individual `<programme>` entries never carry it; `get_live_categories` + each channel's `category_id` is the only source, joined in at ingestion time.
- **`originalAirDate` confirmed unpopulatable for this provider** — checked exhaustively across the entire real dump (grepped for `<episode-num>`, `<date>`, `<category>`, `<sub-title>`: zero occurrences in ~101k programme entries), not just inferred from a small sample. `rules.excludeReruns` stays a documented no-op, not a bug, until a provider that exposes this appears.
- **Real-world data isn't clean — two defensive fixes came directly from a real run, not speculative hardening:** (1) at least one real entry had `stop <= start`, which failed `epg_programs`' own CHECK constraint and — since a provider's refresh is one transaction — rolled back every valid row along with it; now skipped and counted instead. (2) ~2500 entries collided on the same `(channelId, startTime)` slot; now deduped into a `Map` before insert (last-write-wins) instead of hitting the unique index mid-transaction. Both were invisible until tested against real data; a synthetic fixture would not have caught either.
- **Refresh strategy: full delete + reinsert per provider, not upsert** — a complete dump every cycle, so a slot no longer present (rescheduled, channel pulled) should actually disappear; upsert alone can't do that. Batched in chunks of 500 rows per INSERT (a single statement for ~100k rows would exceed SQLite's bound-parameter limit).
- Verified end-to-end against the real "sonix" provider (not a synthetic fixture): 98,697 programmes landed correctly (1,629 distinct channels, categories populated on every row, coherent sequential per-channel schedules, `channel_id` values shaped like real `stream_id`s) after skipping 256 invalid-window and deduping 2,490 duplicate-slot entries — cross-checked twice (a standalone diagnostic script and the real server boot tick) to rule out one particular false alarm: an early "hang" during testing turned out to be a stray leftover process from a previous test run still squatting on the scratch port, not a bug in this code — the actual pipeline completes in ~3.5s end to end (connection resolve → 3 HTTP calls → parse → transaction) once isolated properly. Testing used throwaway iptv-recorder client keys (`db:seed-client`, revoked afterward) and a scratch `DB_PATH`, never the real dev database or the user's own saved connection.
- Not yet done: rule matching against this cached data (TODO2, separate), and behavior on a *second* provider with a differently-shaped Xtream response — this parser's assumptions (bulk XMLTV supported, `epg_channel_id` present and distinct from `stream_id`, no rerun data) are verified against one real provider, not guaranteed universal.

**`channels` table added 2026-07-22** (`server/src/db/schema.ts`, populated in `../epg/refresh.ts`, exposed via `GET /channels?providerId=` in `server/src/routes/channels.ts`) — for the web UI's channel/category pickers, ahead of building them. Separate from `epg_programs` because a channel's name/category is stable metadata that shouldn't depend on a programme currently being cached for it. `fetchChannels`/`fetchCategories` were already being called during the refresh tick for the id-translation step; this just also persists the result (full delete+reinsert per provider, same as `epg_programs`) instead of discarding it afterward. Required adding a `name` field to `fetchChannels`'s return shape (`server/src/epg/xtream.ts`) — not needed for id-translation, only for display. No pagination/search endpoint yet — verified live with 4,518 real channels across 69 real categories; the UI is expected to filter client-side over one fetched list until that's not fast enough.

**Fixes from real usage, 2026-07-22** — the user tested the rule builder directly (screenshots pushed from a Windows box, phobos, via `scp` to docker-server since drag-and-drop into chat only copies a file path, not the image itself — read locally from there) and found three real issues no amount of curl testing surfaces:
- **"Genre" renamed to "category" throughout** — direct correction: "category" is the actual IPTV/Xtream term (the Xtream API itself calls this `category_name`/`category_id`), "genre" was never anything but this codebase's own wrong guess. Renamed everywhere: `rules.categories`/`channels.category`/`epg_programs.category` columns, the matcher, every API field and response shape, all UI labels. The DB rename used a hand-verified `ALTER TABLE ... RENAME COLUMN` migration rather than trusting drizzle-kit's default — its interactive rename-vs-recreate prompt, when driven non-interactively, generated a `rules` table rebuild whose `INSERT ... SELECT` referenced the *new* column name (`categories`) against the *old* table (which still only had `genres`); SQLite's double-quoted-identifier-falls-back-to-string-literal quirk meant this didn't error, it silently overwrote every rule's real filter data with the literal text `"categories"`. Caught by testing the migration against a copy of the real DB with an actual data row first, not just checking that the migration ran without throwing.
- **Live preview showing matches "as if a channel were selected" when none was** — not actually a bug (a category-only rule correctly matches every channel in that category, by design), but the *channel picker* not narrowing itself down was confusing: selecting a category didn't filter which channels showed up in the channel search box below it, so the two pickers felt disconnected. Fixed: selecting one or more categories now narrows the channel list to just those categories' channels (verified live: 4,518 channels shown with none selected, 5 shown after selecting "US| BIBLICAL/RELIGIOUS", matching that category's real channel count). Deliberately one-directional — narrowing never deselects a channel already explicitly picked, even if a later category change would have excluded it from the visible list.
- **Layout overflow, found by looking at an actual screenshot, not just a curl-verified API:** the live preview panel rendered ~170px past the right edge of the page. Root cause: `<select multiple>` elements had no explicit width, so they sized to their widest option text (real channel/category names can be long), which forced the whole form column wider than its grid share. Fixed with explicit `width: 100%` on form inputs/selects plus `min-width: 0` on the grid items (a classic CSS grid gotcha — items default to `min-width: auto`, so intrinsic content size can still overflow a fr-sized track). That same width fix initially over-corrected and stretched the checkboxes too, shoving their labels far to the right — caught in the very next screenshot and scoped the width rule to exclude `input[type="checkbox"]`.

**More fixes from real usage, 2026-07-22 (same session, next screenshot):**
- **Matches no longer include past airings.** The live preview and `GET /rules/{id}/matches` were returning every matching row regardless of whether it had already aired — direct feedback: "I'm not sure how having shows from the past would be helpful." A program whose start time has passed isn't something a fresh `POST /recordings` call could ever schedule (iptv-recorder schedules a start/end window, not "join a stream already in progress"), so `matchRule()` (`server/src/rules/matcher.ts`) now always requires `startTime` in the future — pushed down as a SQL condition (`gt(epgPrograms.startTime, new Date())`), not a JS-side filter, applying to both endpoints since both call the same function. Verified: a `seriesTitle: "The Big Bang Theory"` rule dropped from 729 matches (all-time) to 320 (future-only), exactly matching a raw `COUNT(*) WHERE start_time > now`.
- **New: hide channels with no upcoming EPG data, in the channel picker.** Direct feedback: "there are many channels with no EPG at all... they won't lend themselves to setting up a scheduled recording." `GET /channels` (`server/src/routes/channels.ts`) now computes a `hasEpg` boolean per channel (an `EXISTS` subquery against `epg_programs` with the same future-only cutoff as the matcher, for the same reason — a channel whose only cached programs are already in the past is exactly as useless to pick as one with none). A checkbox in `RuleForm.tsx`, checked by default, hides channels with `hasEpg: false` from the picker (2,891 of the real 4,518 channels, verified live) — deliberately non-destructive like the category-narrowing filter: it never deselects a channel already explicitly picked.
- **A real, instructive SQL bug in that `hasEpg` query:** the first version wrote the correlated subquery using drizzle's typed column interpolation (`` sql`${epgPrograms.providerId} = ${channels.providerId}` ``), which seemed reasonable but is wrong for this case — `${col}` only ever emits the column's bare quoted name with no table prefix, so inside the subquery both sides rendered as the identical unqualified `"provider_id"`, comparing `epg_programs.provider_id` to itself rather than correlating against the outer `channels` row. Every channel came back `hasEpg: true` — passed a shallow smoke test, but cross-checking against the already-known real count (~1,627 of 4,518 channels actually have cached programs) immediately showed it was wrong. Fixed by writing the correlated columns as plain table-qualified SQL text (`epg_programs.provider_id = channels.provider_id`) instead of drizzle's column interpolation. Re-verified against the same ground-truth count until it matched exactly (1,627/4,518).

## Minimal Rule Execution

**Implemented 2026-07-22** — the step where iptv-scheduler starts doing its actual job: turning a rule's live matches into real `POST /recordings` calls against iptv-recorder, rather than only ever previewing them.

- **`scheduled_recordings`** (`server/src/db/schema.ts`) — a dedup ledger, one row per successfully-submitted recording. Keyed on `(providerId, channelId, startTime)` — the same natural key `epg_programs` itself uses — deliberately *not* including `ruleId`: if two different rules both match the same real airing, it's still only ever scheduled once. A rejected submission (iptv-recorder 400/404/409) gets no row and is simply retried next tick.
- **`execution_config`** — singleton, mirrors `recorder_config`'s pattern: `{ automaticSchedulingEnabled (bool, default false) }`, exposed via `GET`/`PUT /config/execution` and a checkbox on the Settings page ("Automatically schedule matching recordings"). Off by default — this is the first feature that can make iptv-recorder actually start recording, using real disk space and real concurrent-stream slots, so the execution tick runs on its own timer regardless but does nothing until this is explicitly turned on.
- **`server/src/scheduling/execute.ts`/`index.ts`** — `runScheduleExecution()`, an in-process interval tick (`EXECUTION_INTERVAL_MS`, default 15 minutes — finer than EPG's 6-hour refresh since a rule match becoming schedulable is time-sensitive, coarser than iptv-recorder's own 30s dispatch tick), same lifecycle shape as `epg/index.ts`. Each tick: reads `execution_config` (no-op, no log, if off); fetches enabled rules ordered `priority DESC` (the first time this column has actually been consumed — see PLAN.md Rule Schema, "carried on the rule now... ahead of the conflict-resolution logic that will actually consume it"); reuses `matchRule()` (`server/src/rules/matcher.ts`) as-is, no new matching logic; loads the tick's already-scheduled slots into an in-memory `Set` up front (covers both prior ticks and de-dupes within the same tick); calls the new `scheduleRecording()` for each not-yet-scheduled match.
- **A real correctness point caught by review before it became a bug, not found by testing:** a match must be scheduled using **the match's own `providerId`/`channelId` (the `epg_programs` row), not the rule's** — `rules.providerId` is nullable (a rule can match across every configured provider), but `epgPrograms.providerId` is `NOT NULL` and is the actual provider a given airing belongs to. Using `rule.providerId` would have silently broken any provider-unscoped rule the first time a second provider existed.
- **A real control-flow bug caught the same way:** the tick needs exactly one outer `try`/`catch` around the whole rule × match double loop, not one per match. `scheduleRecording()` returns a non-throwing `{ok:false}` for a business-level rejection (400/404/409 — all treated uniformly, not just 409) and only *throws* for a genuine connectivity failure or `RecorderNotConfiguredError`; wrapping each match's call in its own `try`/`catch` would have silently confined a thrown "abort the tick" signal to just the current rule's remaining matches (a `break` doesn't cross an outer loop boundary), continuing on to the next rule instead of actually aborting. Also added a `ticking` reentrancy guard, absent from the EPG refresh precedent this otherwise mirrors — this tick does one live network round-trip per match rather than one call per provider, making an overlapping tick more plausible if iptv-recorder is slow.
- **`server/src/recorderClient.ts`**: new `scheduleRecording()` deliberately bypasses the existing `rawFetch`/`recorderFetch` helpers (which always throw on a non-2xx response) and does its own `fetch()` after the existing `requireConnection()`, branching on `response.status` itself — 201 is success, anything else non-throwing is `{ok:false, status, error}`. `rawFetch`/`recorderFetch`/`testRecorderConnection` are untouched.
- **A genuine, live-tested surprise about foreign keys in this codebase:** `scheduled_recordings.ruleId` was originally written as `integer("rule_id").notNull().references(() => rules.id)`, on the assumption (checked against `server/src/db/client.ts`, which never calls `PRAGMA foreign_keys = ON`) that FK enforcement was off project-wide, matching every other table's opaque, unenforced `providerId` columns. Caught live during test cleanup: deleting a rule that still had a `scheduled_recordings` row pointing at it failed with `SQLITE_CONSTRAINT_FOREIGNKEY` — **better-sqlite3 turns `PRAGMA foreign_keys` on by default per connection** (confirmed directly: a fresh in-memory `better-sqlite3` handle reports `foreign_keys = 1`), unlike the plain `sqlite3` CLI used to inspect the file, which reports `0` — the CLI check that seemed to confirm "FKs are off here" was checking the wrong thing's default. This would have silently broken the already-decided "no cascade" behavior (PLAN.md: "deleting a rule doesn't retroactively cancel anything already scheduled") by making rule deletion *fail outright* instead of leaving history alone, the first time a rule with any scheduled history was ever deleted. Fixed: `ruleId` is a plain `integer(...).notNull()` with no `.references()` at all — verified against a scratch copy of the real DB (insert a rule, point a `scheduled_recordings` row at it, delete the rule with `PRAGMA foreign_keys=ON`) that deletion now succeeds and the historical row survives with its original `ruleId` intact.
- **Migration safety, standard process for this project:** both new tables generated as clean additive `CREATE TABLE`s (migration `0004`); the follow-up FK removal (migration `0005`) was, as expected, a drizzle-kit table-rebuild (`__new_scheduled_recordings` + `INSERT ... SELECT` + rename) — the exact shape that corrupted real data once before in this project (see Web UI: Rules, the category rename incident) — so it was tested against a scratch copy with an actual row inserted first, confirming the row and its data survived the rebuild intact and the FK constraint was actually gone, before applying either migration to the real dev DB.
- Verified end-to-end against the real "sonix" provider (`maxConcurrentStreams: 1`): with the toggle off, the tick fires silently with zero `POST /recordings` calls and an empty `scheduled_recordings` table. With it on, two narrowly-scoped test rules (one matching a real near-future airing on one channel, a second matching a real near-future overlapping airing on a different channel — same provider, whose 1-stream cap made a genuine rejection trivial to trigger without any synthetic setup) produced exactly the expected outcome: `scheduled=3 alreadyScheduled=0 rejected=1`, with the rejected one logging the real `409` reason (`would exceed provider's max concurrent streams`); the real recordings and `recorderRecordingId`s were confirmed via iptv-recorder's own `GET /recordings`; re-running the tick produced `scheduled=0 alreadyScheduled=3 rejected=1` (dedup holds, rejection keeps retrying rather than being silently dropped). All three real recordings were immediately cancelled via `DELETE /recordings/{id}` rather than left to run.
- **A real bug found in iptv-recorder, not iptv-scheduler, during this cancellation step:** `DELETE /recordings/:id` (`iptv-recorder/server/src/routes/recordings.ts:505,511`) writes `return reply.code(204);` with no `.send()` — the database write (soft-cancel or hard-delete) completes correctly and near-instantly, but the HTTP response itself never gets sent, hanging the request indefinitely. Confirmed live: the DB row's `status` flipped to `cancelled` within tens of milliseconds even though the client-side request never returned. Worked around during this cleanup with a bounded client-side timeout per request plus a follow-up `GET /recordings` to confirm each cancellation actually landed; not fixed here since it's iptv-recorder's codebase, not this one — flagged for a fix there. **Fixed in iptv-recorder 2026-07-22, commit `e5e095f`** — both branches now call `.send()`; the same pattern was also found and fixed in `DELETE /providers/:id`, and every other `reply.code(...)` call site was audited with no further instances found.
- Web UI: a single checkbox added to `web/src/pages/Settings.tsx`, using the `api.get`/`api.put` wrapper (not `RecorderSettings.tsx`'s older raw-`fetch` style). Verified in a real headless-Chromium session (Playwright, installed temporarily per this project's precedent, removed after): default unchecked (matches the DB default), checking it and reloading the page preserves the state (proves it's the DB row, not React state), unchecking restores default, zero console errors.
- All test rules, ledger rows, and the toggle itself were cleaned up/reset after verification, leaving the real dev DB in the same state it started in.

## Conflict Resolution / Preemption (TODO3)

**Implemented 2026-07-23** — the execution tick has always used `iptv-recorder 409` rejections as a dead end: log it, retry next tick, forever (PLAN.md "Minimal Rule Execution"). `rules.priority` existed and was already consumed for iteration order, but never for actually resolving a conflict. This is the first feature that lets iptv-scheduler cancel a real, already-scheduled recording on its own — user confirmed a separate, off-by-default opt-in toggle for that reason (see below) before any of this was built.

- **Only two of iptv-recorder's four hard-reject reasons** (`server/src/hardReject.ts` in the iptv-recorder repo, read directly rather than assumed) **are actually preemptable:** `conflicts with an existing recording on this channel` and `would exceed provider's max concurrent streams`. `provider is disabled` and `insufficient storage space` are capacity-independent — no cancellation fixes either — so those two keep behaving exactly as before (log, retry next tick). `isPreemptable()` (`server/src/scheduling/preempt.ts`) gates on this.
- **New `execution_config.preemptionEnabled`**, deliberately a separate toggle from `automaticSchedulingEnabled` rather than bundled in — confirmed with the user before building: a checkbox that can make this app cancel a recording nobody asked it to cancel is a materially bigger blast radius than one that only ever creates new ones, even though preemption only ever does anything while automatic scheduling is also on (Settings UI disables the checkbox until the other one is checked). Migration `0007`, a clean additive `ALTER TABLE ... ADD ... DEFAULT false NOT NULL` — no rename ambiguity, same shape as `includeInProgress`'s addition.
- **`server/src/scheduling/preempt.ts`, `attemptPreemption()`** — on a preemptable rejection, finds every other *active* (scheduled/recording) recording on that provider overlapping the candidate's window (`listRecordings({ providerId })`, a new optional filter on the existing `listRecordings()`), cross-references each against the local `scheduled_recordings` ledger to find which rule (and priority) scheduled it, and restricts candidates to **a known ledger row with a known rule whose priority is strictly lower than the rule currently trying to schedule.** A manual booking (`ruleId: null`) or any recording iptv-scheduler doesn't recognize at all (booked directly through iptv-recorder, outside this app) is never touched, and equal priority never preempts — first-in-wins, the same principle iptv-recorder's own `hardReject.ts` already applies to same-channel conflicts. Candidates are cancelled lowest-priority-first via the existing `cancelRecording()` (the same soft-cancel path the Recordings page's own `Cancel` button uses), re-simulating locally (a reimplementation of iptv-recorder's own sweep-line `maxConcurrentOverlap`) after each cancellation to stop as soon as the original request would no longer be rejected, rather than over-cancelling.
- **A preempted recording's local ledger row is deliberately left untouched** — same "soft-cancel keeps the row" rule from "Web UI: Recordings" above applies automatically, with no extra code needed for it. This means a preempted recording just shows up on the Recordings page exactly like a user-cancelled one: `cancelled` status, and a `Reactivate` button if its window hasn't passed — full visibility and an easy undo, for free, from a feature built for an unrelated reason two commits earlier.
- **`server/src/scheduling/execute.ts`**: on `!result.ok && config.preemptionEnabled && isPreemptable(...)`, looks up the match's provider (`listProviders()`, only fetched at all when `preemptionEnabled` — no wasted call otherwise) for its `maxConcurrentStreams`, calls `attemptPreemption()`, and retries the original `scheduleRecording()` call exactly once if anything was actually freed. Each preemption is logged with both rules' identities and the freed recording's id. **Idempotency comes for free, not from new code:** a slot's `scheduled_recordings` row (cancelled or not) is still in the tick's `seen` Set, so a slot that already went through a preempt-and-reclaim cycle is never re-attempted or re-fought-over on a later tick — verified live, see below.
- Verified end-to-end against real data on the real "sonix" provider (`maxConcurrentStreams: 1`): a low-priority rule (priority 1) scheduled 11 real recordings on one channel across the guide's ~14h lookahead; a high-priority rule (priority 10) then created on a different, time-overlapping channel correctly preempted all 11 — each log line correctly attributing the cancelled recording to rule 7 by id/name, each low-priority ledger row flipping to `cancelled` and each high-priority one landing as `scheduled` for the same slot (`scheduled=11 alreadyScheduled=11 preempted=11 rejected=6`). The 6 rejections were the real, already-existing "NFL Football" manual booking's overlap window — correctly left alone every single time (manual bookings are never preemptable) rather than incorrectly preempted. A third tick against the same state produced `scheduled=0 alreadyScheduled=22 preempted=0 rejected=6` — confirming no thrashing between the two rules and that the permanently-unresolvable NFL overlap keeps retrying forever exactly like a genuine non-preemptable rejection always has. All test rules and ledger rows were cleaned up afterward through the app itself (cancel-then-delete, per "Web UI: Recordings"'s own two-step terminal-deletion behavior); both toggles were left off; the user's real NFL Football recording was never touched.

- **Web UI** (`web/src/pages/ScheduledRecordings.tsx`, new nav entry "Recordings" in `web/src/App.tsx`): a table (title, channel, rule-or-"Manual", start, end, status badge, action button). **Reactivate, added same day** — direct follow-up, user feedback right after the cancel/delete split above landed: "I like that the row is still there [after cancel]. But the fact that the only thing you can do from there is delete[] is less helpful... if we can't reactivate it, then what's the point in keeping it around?" iptv-recorder has no way to un-cancel a specific recording — its `DELETE /recordings/:id` only ever sets `status: "cancelled"`, never back (checked directly against its code before building this, not assumed) — so "reactivate" means re-submitting the same provider/channel/time window as a brand-new recording via the existing `scheduleRecording()`, and repointing the *same* ledger row at the new `recorderRecordingId`, rather than resurrecting the old one.

- **New `POST /scheduled-recordings/:id/reactivate`** (`server/src/routes/scheduledRecordings.ts`) — guards: 404 if the ledger row doesn't exist, 400 if the original `endTime` has already passed (nothing left to record), 400 if the row's live status isn't currently `cancelled` (checked fresh via `getRecording()`, not trusted from the client). On success, updates the existing row's `recorderRecordingId` in place (same natural key, same row — this is a re-submission of the same slot, not a new booking) and returns it through a new `toDetailResponse()` helper (the single-row equivalent of `GET /scheduled-recordings`' bulk enrichment — two extra one-off queries are cheap for a single row, not worth threading the list route's bulk maps through for).
- **Web UI**: a `Reactivate` button (new `.button-reactivate` CSS, accent-colored, same compact sizing as `.button-danger`) appears only when `status === "cancelled"` *and* `endTime` is still in the future — both checked client-side to decide whether to show the button at all, and re-checked server-side since the client can't be trusted for a state-changing decision. Sits alongside the existing `Cancel`/`Delete` button in a new `.row-actions` flex wrapper, gated behind the same `confirm()` convention as every other destructive-ish action here.

## API Documentation

**Implemented 2026-07-23** — direct follow-up after the user asked what iptv-scheduler's own API surface even looks like (13 routes across 4 files existed by this point, more than expected since this app is the "brain" half of the split with iptv-recorder — EPG/rules/config/scheduling state, not a thin proxy). Mirrors iptv-recorder's own Swagger setup ([its PLAN.md](/srv/iptv-recorder/PLAN.md), done 2026-07-19) almost exactly, since that pattern was already proven there.

- `@fastify/swagger` (openapi mode) + `@fastify/swagger-ui`, registered in `server/src/index.ts` right after the app is constructed, served at `/documentation` (UI) and `/documentation/json`/`/documentation/yaml`.
- **Deliberately no `bearerAuth`/`security` block**, unlike iptv-recorder's version — this app has no auth at all yet (see Open Questions below), so there's nothing to declare or exempt `/health` from. `/documentation` itself is unauthenticated as a result, same as every other route.
- Reused iptv-recorder's `refResolver.buildLocalReference` fix directly — Fastify's default resolver names every shared `$ref` component `def-0`, `def-1`, ... and stashes the schema's real `$id` in `title` instead, which would make the generated doc's component list unreadable without it.
- Every one of the 14 operations across all 5 route files (including the two bare `/health`/`/health/db` checks, which had no schema object at all before this) got `tags` (one per resource group: `health`, `config`, `channels`, `rules`, `scheduled-recordings`) and a `summary`; `description` was added selectively — only where a route's behavior isn't self-evident from its summary/schema alone (e.g. `PUT /config/recorder`'s test-before-save, `DELETE /rules/:id`'s no-cascade-to-scheduled_recordings behavior, `POST /scheduled-recordings`'s pass-through-of-iptv-recorder's-own-rejection behavior) — not exhaustive prose on every route, matching iptv-recorder's own ratio.
- Verified end-to-end: `GET /documentation/json` returns all 7 `$id`-registered components (`Rule`, `Channel`, `MatchedProgram`, `ScheduledRecording`, `RecorderConfig`, `ExecutionConfig`, `Error`) under their real names, and all 10 unique paths with correct tags/summaries. In a real headless-Chromium session (Playwright, installed temporarily and removed after): the UI actually renders (title "Swagger UI"), all 5 tag sections appear, and a real "Try it out" → "Execute" against `GET /rules` returned a genuine `200` with a rendered response body — confirming the UI is actually usable, not just that the JSON document is well-formed.
